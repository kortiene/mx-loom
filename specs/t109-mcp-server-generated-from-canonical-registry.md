# T109 · binding: MCP server generated from the canonical registry

> GitHub issue **#17** · `area/mcp` `type/feature` `P0` · Estimate **L** · Milestone **M1 — Delegation MVP**
> Source: `docs/backlog.md` (`T109`). Blocked-by **#11/T103** (deferred-result protocol), **#12/T104** (discovery handlers), **#13/T105** (delegation handler) — all **landed**. This is a planning/spec document only; it does **not** implement the feature.

## Problem Statement

mx-loom's locked design decision #1 is "both bindings in parallel": the one canonical, transport-neutral tool registry must feed **both** a generated **MCP server** (the universal binding every target runtime — Claude/ADK/OpenCode/Pi/custom — can speak) **and** a Claude Agent SDK native shim, from day one (`docs/mx-agent-tool-fabric-design.md` §3, §8, locked decision 1).

Today the registry half is complete: `@mx-loom/registry` exposes the nine canonical `mx_*` descriptors (`CANONICAL_M1_TOOLS`), the normalized T102 result envelope + constructor helpers, and the nine handlers (`mxFindAgents`, `mxDescribeAgent`, `mxDelegateTool`, `mxRunCommand`, `mxAwaitResult`, `mxShareContext`, `mxGetContext`, `mxCancel`, `mxWorkspaceStatus`) wired to an injected daemon-call seam. `@mx-loom/toolbelt` provides the transport (`MxClient`/`createClient`) and the session model (`openSession` → `MxSession`). `@mx-loom/claude` provides the JSON Schema → Zod converter (T111) that the *Claude* shim (T110) will use.

**The gap:** there is no module that exposes these descriptors + handlers to an external runtime over the Model Context Protocol. No `tools/list` advertising the nine `mx_*` tools with their JSON Schemas; no `tools/call` routing a tool invocation to the matching handler; no stdio or remote transport; no mapping of the T102 envelope (especially the `awaiting_approval` deferred status) onto an MCP `CallToolResult`. Until this lands, no MCP-speaking runtime can discover, delegate to, or coordinate with the mesh — and the downstream M2 bindings (T201 ADK `MCPToolset`, T203 OpenCode `mcp` entry, T204 Pi), the golden test's MCP arm (T114), and publication (T602) are all blocked.

T109 closes that gap: a generated MCP server that turns the canonical registry into a live MCP endpoint, secret-free, leaving every enforcement decision out-of-process on the receiving mx-agent daemon.

## Goals

- **AC1 — list:** an MCP client connected to the server lists all nine `mx_*` tools (`CANONICAL_M1_TOOLS`) with correct names, descriptions, and **input JSON Schemas passed through verbatim** from the descriptors (no lossy Zod round-trip).
- **AC2 — round-trip:** a delegated call (`mx_delegate_tool`) issued by an MCP client routes through the server to the `mxDelegateTool` handler, reaches the daemon via the injected `MxClient`/`MxSession` seam, and returns a normalized T102 envelope serialized as an MCP `CallToolResult`.
- **AC3 — approval surfaces:** an `awaiting_approval` result (and its sibling `running`) surfaces correctly over MCP as a non-error, structured result carrying `status`, `handle`, and `approval` — so the model can keep working and later resolve via `mx_await_result(handle)`.
- **Two transports:** the same server runs over **stdio** (local subprocess — the ADK/OpenCode/Claude-external default) and a **remote** transport (Streamable HTTP), selectable without changing tool behavior.
- **One generation path:** tools are generated by *enumerating the registry*, never hand-authored per tool. Adding a tenth descriptor to `CANONICAL_M1_TOOLS` (plus its handler) must surface it over MCP with no edit to the server's per-tool code.
- **Envelope fidelity:** all five envelope statuses (`ok` / `running` / `awaiting_approval` / `denied` / `error`), the closed `error.code` taxonomy, `handle`, `approval`, and `audit_ref` survive the MCP boundary intact and machine-readable.
- **Secret-free chokepoint:** the server holds no Matrix tokens, no signing keys, no provider keys, no `GH_TOKEN`; it routes every daemon call through the toolbelt `MxClient` so the deny-by-default env allowlist, outbound credential-shaped-arg rejection, and inbound `redactSecrets` all stay in force, unmodified.
- **Audit chokepoint:** the server is the single result-return chokepoint where the T113 `withAudit` tap is applied once (best-effort, `NullAuditSink` by default), so every tool result can mirror to the queryable index without the server re-implementing audit.

## Non-Goals

- **Runtime-specific shims (M2).** No ADK `LongRunningFunctionTool` wrapping (T202), no Claude `canUseTool` HITL hook or in-process `createSdkMcpServer` registration (T110), no Pi native-tool map (T204). Those consume *this* generic MCP server (or, for Claude, build a parallel in-process shim). T109 ships only the generic, runtime-agnostic MCP surface.
- **Approval/trust/policy mutation tools.** The model never gets `trust.*`, `approval.decide`, `policy.*`, `auth.*`, `device.*`, `daemon.*`. The MCP server surfaces only the nine model-facing verbs in `CANONICAL_M1_TOOLS`; approval reaches the model purely as the `awaiting_approval` *status*.
- **The Claude binding's Zod converter.** Generic MCP advertises raw JSON Schema; the T111 converter is for the Claude SDK's `tool()` only and is not used here.
- **MCP `resources` / `prompts` / `sampling` / `roots` / `elicitation`.** v1 is `tools`-only. Resources for shared-context fetch are a possible later refinement (noted under Open Questions), not in scope.
- **Streaming tool output into the model.** Per design §9, no `StreamChunk` plumbing; deferred results resolve via `mx_await_result`, artifacts via `mx_get_context`.
- **Multi-tenant / per-tenant auth on the remote endpoint, billing, cost guardrails.** Single workspace, single tenant (design §8). The remote transport ships with a localhost-bind default and a documented hardening checklist, not a full authN/authZ layer.
- **Live two-daemon conformance certification.** Wire-shape assumptions (`call.start` param names, `CallResponse` disposition vocabulary) remain pinned at the existing `MXL_CONFORMANCE_TWO_DAEMON=1` round-trip the handlers already stage; T109 inherits, does not re-pin, them. End-to-end certification is T114.

## Relevant Repository Context

The stack is TypeScript (pnpm workspace, Node ≥20.19, vitest, Apache-2.0, ESM, `nodenext`, `verbatimModuleSyntax`, `strict` + `noUncheckedIndexedAccess`). The repo is **no longer docs-only** — four packages exist under `packages/*` (`pnpm-workspace.yaml` globs `packages/*` + `adw_sdlc`):

- **`@mx-loom/registry`** (`packages/registry`) — the canonical source T109 consumes. Exports (`src/index.ts`):
  - **Descriptors:** `CANONICAL_M1_TOOLS: readonly ToolDescriptor[]` (the nine verbs, in stable order, deep-frozen) and each individual const. A `ToolDescriptor` (`src/descriptor.ts`) is pure secret-free metadata: `{ name, description, input_schema: JsonSchema, output_schema: JsonSchema, async_semantics: 'sync' | 'deferred' }`. `input_schema`/`output_schema` are draft-07 JSON Schema (`JSON_SCHEMA_DIALECT` from `src/validator.ts`). `mx_delegate_tool`'s `input_schema` is the OUTER envelope (`agent`/`tool`/`args` with `args` an open object); the inner-tool schema is validated dynamically by the T105 handler, never baked in.
  - **Envelope (T102):** `ToolResult` = `{ status, result, error, handle, approval, audit_ref }`; constructor helpers `ok`/`running`/`awaitingApproval`/`denied`/`errored`; `ENVELOPE_SCHEMA` + `validateEnvelope`; the closed taxonomy `ERROR_CODES`/`DENIAL_CODES`/`FAULT_CODES` with `mapTransportError`/`mapDaemonError`; `newIdempotencyKey`/`IDEMPOTENCY_KEY_PREFIX`.
  - **Handlers (T103–T108):** `mxAwaitResult`, `mxFindAgents`, `mxDescribeAgent`, `mxDelegateTool`, `mxRunCommand`, `mxShareContext`, `mxGetContext`, `mxCancel`, `mxWorkspaceStatus`. **Every handler has the shape `(input, deps) => Promise<ToolResult>`, never throws, and builds its envelope only through the T102 helpers.** They differ only in their `deps` subtype:
    - `HandlerDeps` (read/handle-only): `mxAwaitResult`, `mxFindAgents`, `mxDescribeAgent`, `mxCancel`.
    - `RoomScopedDeps` (= `HandlerDeps` + optional `room`): `mxShareContext`, `mxGetContext`, `mxWorkspaceStatus`.
    - `DelegateDeps` (= `RoomScopedDeps` + optional `validator`): `mxDelegateTool`.
    - `ExecDeps` (= `RoomScopedDeps`): `mxRunCommand`.
    - The injected daemon seam is `DaemonCall = Pick<MxTransport, 'call'>` (`src/handlers/deps.ts`) — a structural subset of the toolbelt transport, imported `type`-only, so the registry has **zero runtime dependency** on the toolbelt.
  - **Security:** `MODEL_FACING_ALLOWLIST`, `FORBIDDEN_AUTHORITY_VERBS`, `isForbiddenAuthorityVerb`, `CREDENTIAL_KEY_RE`, `findCredentialShapedProperty` — the no-authority allowlist + secret-free-shape oracle.
  - Runtime dep: `ajv@^8.20.0`. Dev-only dep on `@mx-loom/toolbelt`.

- **`@mx-loom/toolbelt`** (`packages/toolbelt`) — the transport + session. Exports `MxClient`/`createClient` (`MxTransport` with `call(method, params?, options?)`, IPC-primary + CLI fallback), `openSession`/`MxSession` (registers once via `agent.register`, runs a heartbeat, threads a session-stable `correlation_id`, exposes `call(method, params?, options?)`, `agentId`, `room?: string`, `correlationId`, `close()`), and the secret-boundary guards `assertNoCredentialShapedArgs` (outbound, pre-dispatch) + `redactSecrets` (inbound, on the `MxClient.call` seam). **`MxClient` / `MxSession` are the concrete `DaemonCall`** the handlers' injected seam expects.

- **`@mx-loom/claude`** (`packages/claude`) — T111 JSON Schema → Zod converter (`jsonSchemaToZod`/`jsonSchemaToZodRawShape`, fail-closed). Built for the *Claude* shim (T110); **not used by T109**.

- **`@mx-loom/audit`** (`packages/audit`) — T113. `auditRowFrom(result, ctx)` pure projection, `AuditSink` port (`PostgresAuditSink`/`InMemoryAuditSink`/`NullAuditSink`), and the best-effort **`withAudit` tap a binding applies once at its single result-return chokepoint** (explicitly staged "for T109/T110"). Default off; live Postgres gated behind `MXL_AUDIT_PG=1`.

**The MCP SDK is already resolvable.** `@modelcontextprotocol/sdk@1.29.0` is present in `pnpm-lock.yaml` (pulled transitively by `@anthropic-ai/claude-agent-sdk` and the Pi SDK). T109 should declare it as a **direct** dependency of the new package and pin to that line (`^1.29.0`).

**What does not exist yet (decisions to confirm, not assume built):**
- **No MCP binding package.** There is no `packages/mcp` / `@mx-loom/mcp`. Its name, location, and exact public surface are proposed below and are decisions to confirm.
- **No name→handler dispatch table.** The nine handlers are exported individually; there is no central map from a descriptor `name` to its `(input, deps)` handler. T109 must build one (the registry exposes `CANONICAL_M1_TOOLS` for listing but no execution router).
- **No envelope→`CallToolResult` serializer.** The mapping of `ToolResult` onto MCP's content/`structuredContent`/`isError` shape is new code.
- **No server bootstrap / lifecycle / CLI entry.** Opening the `MxSession`, choosing a transport, and shutting down cleanly are new.

Package scaffolding convention (mirror `packages/claude`): `package.json` (`"type": "module"`, `exports: { ".": "./src/index.ts" }`, scripts `typecheck`/`build`/`test`), `tsconfig.json` (the strict/nodenext base) + `tsconfig.build.json` (emit to `dist`), `vitest.config.ts`, `src/`, `test/`, `README.md`.

## Proposed Implementation

Create a new leaf package **`@mx-loom/mcp`** at `packages/mcp` that depends on `@mx-loom/registry` (descriptors + handlers + envelope), `@mx-loom/toolbelt` (transport/session — the concrete `DaemonCall`), `@mx-loom/audit` (the `withAudit` tap), and `@modelcontextprotocol/sdk` (the protocol). Five concerns, each a small module:

### 1. The tool generator (`src/tools.ts`) — descriptors → MCP tool list

Enumerate `CANONICAL_M1_TOOLS` and produce the MCP `Tool[]` for `tools/list`. Each MCP tool is `{ name, description, inputSchema, outputSchema?, annotations? }` where `inputSchema` is the descriptor's `input_schema` **passed through verbatim** (it is already draft-07 JSON Schema — exactly what MCP's `inputSchema` expects). This is the reason to use the SDK's **low-level `Server`** (`@modelcontextprotocol/sdk/server/index.js`) with `server.setRequestHandler(ListToolsRequestSchema, …)` rather than the high-level `McpServer.registerTool`, which is Zod-first and would force a lossy JSON Schema → Zod → JSON Schema round-trip. Low-level keeps the registry's JSON Schema as the single source of truth (Goal "one generation path", AC1).
- Map `output_schema` → MCP `outputSchema` (also verbatim). Where the descriptor's output is an open object (`mx_delegate_tool`, `mx_await_result`), advertise it as such; the server does **not** validate `result` against it (the daemon/handler owns the success payload) — see Open Question on `structuredContent` validation.
- Set MCP tool `annotations` honestly from `async_semantics`: `deferred` tools (`mx_delegate_tool`, `mx_run_command`) carry a hint that they may return a handle; mutating verbs are not `readOnlyHint`. This is advisory metadata only — never an enforcement signal.

### 2. The dispatch table (`src/dispatch.ts`) — name → handler

A single typed map from each descriptor `name` to a closure `(input, deps) => Promise<ToolResult>`, built once over the nine handlers. Each entry knows which `deps` subtype its handler needs and is constructed from a shared `BindingContext` (below). Because all nine handlers share the `(input, deps) => Promise<ToolResult>` shape and never throw, the router is uniform: look up by name, call, get a `ToolResult`. An unknown tool name → a constructed `errored('not_found', …)` envelope (the model asked for a tool we do not surface), never a thrown exception.
- `mx_delegate_tool` gets `DelegateDeps` (validator defaulted by the handler), `mx_run_command`/`mx_share_context`/`mx_get_context`/`mx_workspace_status` get `RoomScopedDeps` (room from the session), the read/handle verbs get `HandlerDeps`. The `daemon` field of every deps object is the same injected `DaemonCall`.

### 3. The binding context + session wiring (`src/context.ts`)

`BindingContext` holds: the `DaemonCall` (a live `MxSession` or bare `MxClient`), the session `room` (from `MxSession.room`, **never** model input — the model never names a Matrix room id), and the `AuditSink` (default `NullAuditSink`). A `createBindingContext(opts)` opens a session via the toolbelt's `openSession` (one `agent.register`, heartbeat, correlation threading) and returns the context plus a `close()` that stops the heartbeat and deregisters. **Prefer the `MxSession` as the `DaemonCall`** so every daemon RPC is correlation-stamped and routed through the credential guard + redaction; fall back to a bare `MxClient` only if a caller supplies one. The context never reads or stores a secret — the toolbelt holds the socket and the env allowlist.

### 4. The envelope serializer (`src/serialize.ts`) — `ToolResult` → `CallToolResult`

The crux of AC2/AC3. Map the T102 envelope onto MCP's `CallToolResult` = `{ content: ContentBlock[], structuredContent?, isError? }`:
- **`structuredContent`** = the **full `ToolResult` envelope** verbatim (`status`, `result`, `error`, `handle`, `approval`, `audit_ref`). This is the machine-readable channel modern MCP clients consume, and it makes every status — including `awaiting_approval` (handle + approval) and `denied` (denial code) — programmatically reactable (design §4.5: "a runtime can react programmatically").
- **`content`** = a single `text` block carrying a human-/model-readable JSON rendering of the same envelope, for clients that do not read `structuredContent`. Never embed a secret (the envelope is already secret-free by the T102/T008 contract).
- **`isError`** = `true` **only** for `status: "error"` (the fault-set: `timeout`/`not_found`/`invalid_args`/`target_offline`/`internal`) — a genuine fault the protocol should flag. **`denied`, `awaiting_approval`, `running`, and `ok` are NOT `isError`**: `denied` is a governance outcome the model must read and adapt to (not a transport failure), and `awaiting_approval`/`running` are legitimate in-progress states. Flagging them as protocol errors would push runtimes to retry or abort instead of awaiting/replanning. This decision is the single most important serialization rule and must be unit-pinned.
- The serializer is pure and total (envelope in, `CallToolResult` out) — testable with no daemon.

### 5. The server assembly + transports (`src/server.ts`, `src/cli.ts`, `src/index.ts`)

- `createMcpServer(ctx): Server` — instantiates the low-level SDK `Server` with name/version, registers the two request handlers: `ListToolsRequestSchema` → generator (§1); `CallToolRequestSchema` → router: look up the handler (§2), build its `deps` from `ctx`, call it, apply the **`withAudit` tap once** here (§ the single chokepoint), then serialize (§4). This is also where a credential-shaped-args concern is naturally covered: the router passes args straight to the handler, whose concrete `deps.daemon` (`MxClient`) runs `assertNoCredentialShapedArgs` before dispatch — the server re-implements nothing.
- **stdio transport:** `StdioServerTransport` from `@modelcontextprotocol/sdk/server/stdio.js`. A `bin` entry (`mx-loom-mcp`) opens the session, connects the server over stdio, and shuts down on `SIGINT`/`SIGTERM`/stdin close. This is the artifact ADK (`MCPToolset`), OpenCode (`opencode.json` `mcp`), and Claude-external register.
- **remote transport:** `StreamableHTTPServerTransport` from `@modelcontextprotocol/sdk/server/streamableHttp.js` behind a minimal Node `http` listener. **Bind to `127.0.0.1` by default**; the bind address/port are explicit opt-in flags. Document that exposing the endpoint beyond localhost requires an operator-supplied reverse proxy with auth — the MCP server itself adds no authN (and even so, the daemon still independently enforces trust/policy/approval, so a reachable endpoint cannot escalate; design §1 governing rule). (Do **not** ship the deprecated HTTP+SSE transport.)
- `src/index.ts` exports the library surface (`createMcpServer`, `createBindingContext`, the generator, the serializer, the dispatch map) so T110/T201/T203/T204/T114 can embed or drive the server programmatically; `src/cli.ts` is the thin transport-selecting entry behind `bin`.

**Async via `mx_await_result` (AC3 mechanics).** Generic MCP has no native long-running-tool protocol (that is ADK's `LongRunningFunctionTool`, M2). So a `deferred` tool that returns `running`/`awaiting_approval` surfaces the `handle` (+ `approval`) in `structuredContent`, the model keeps working, and resolves later by calling the ordinary `mx_await_result` tool with that handle. The server does **not** hide the poll loop (that is the per-runtime shim's job) — it faithfully surfaces the handle, which is exactly what "generic MCP surfaces the handle" means (design §3). The handlers already honor an optional inline `wait_ms`; the server passes it through unchanged and never blocks unboundedly.

## API / Interface Changes

- **New public package API (`@mx-loom/mcp`):** `createMcpServer(ctx)`, `createBindingContext(opts)`, the tool generator, the envelope serializer, and the dispatch map. Document all of these in the package `README.md` and TSDoc.
- **New executable:** `mx-loom-mcp` (`bin`) with flags to select transport (`--stdio` default, `--http --host 127.0.0.1 --port <n>`), the agent registration identity/workspace passed to `openSession`, and an optional `--audit` flag (or `MXL_AUDIT_PG=1`) to switch the sink from `NullAuditSink` to `PostgresAuditSink`.
- **No change to the registry descriptors, the T102 envelope, the handler signatures, the daemon-RPC surface, or the toolbelt transport.** T109 is a pure consumer of existing surfaces.
- **MCP wire surface (new, but standard):** the server speaks MCP `initialize`, `tools/list`, `tools/call` (the SDK handles the framing). Tool names are the nine `mx_*` verbs; `inputSchema`/`outputSchema` are the descriptors' JSON Schema verbatim.

## Data Model / Protocol Changes

- **No change** to the T102 result-envelope shape, the closed nine-code error taxonomy, the descriptor model, the idempotency-key contract, or any audit-row column. T109 *transports and serializes* these; it does not alter them.
- **New, additive serialization mapping** (`ToolResult` → MCP `CallToolResult`), defined entirely in `@mx-loom/mcp`:
  - `structuredContent` ← the full envelope (`status`, `result`, `error`, `handle`, `approval`, `audit_ref`).
  - `content[0]` ← a `text` JSON rendering of the same envelope.
  - `isError` ← `true` iff `status === "error"`.
  - This mapping is a documented invariant, unit-pinned, and stable (downstream M2 bindings and T114 depend on it).
- **Idempotency:** the mutating tools (`mx_delegate_tool`, `mx_run_command`) already declare an optional `idempotency_key` in their `input_schema`; the server passes it through verbatim. It does not generate or strip keys (the handler defaults one when omitted).
- **Audit rows:** unchanged schema; the server is merely the *site* where `withAudit` is applied once per result. The binding-supplied `AuditContext` (tool name, `correlation_id` from the session, `idempotency_key` if present) is assembled here.

## Security & Compliance Considerations

- **Secret boundary (Boundary A) holds.** The MCP server lives in the adaptation plane and is **secret-free by construction**: it never holds or forwards Matrix tokens, Ed25519 signing keys, provider keys, or `GH_TOKEN`. It reaches the daemon *only* through the toolbelt `MxClient`/`MxSession`, which enforce the deny-by-default env allowlist; runner/child processes (if any) receive retrieved TEXT only, never credentials. The server starts no child that could inherit a secret-bearing env.
- **No tool field carries a credential, inbound or outbound.** The server re-implements no scrubbing: every daemon call routes through `MxClient.call`, which runs `assertNoCredentialShapedArgs` before dispatch (a credential-shaped `args` key surfaces as `invalid_args`) and `redactSecrets` on the inbound result. The registry's `findCredentialShapedProperty`/`CREDENTIAL_KEY_RE` remain the secret-free-shape oracle. The serializer must never copy anything outside the already-secret-free envelope into `content`/`structuredContent`.
- **Out-of-process enforcement is untouched.** Trust (Ed25519 store), deny-by-default `policy.toml`, sandbox, and human approval gates all run on the *receiving* mx-agent daemon. The MCP server **maps** `policy_denied` / `untrusted_key` / `awaiting_approval` / `approval_denied` / `approval_expired`; it never decides them. Cognition (the connected runtime) can only produce a signed request; the MCP server cannot grant authority and exposes no surface to do so.
- **No authority tools surfaced.** The generator iterates `CANONICAL_M1_TOOLS` only; `trust.*`, `approval.decide`, `policy.*`, `auth.*`, `device.*`, `daemon.*` are structurally unreachable. A regression test asserts the listed tool set is exactly the nine model-facing verbs and intersects `FORBIDDEN_AUTHORITY_VERBS` at the empty set. Approval reaches the model only as the `awaiting_approval` *status*, re-validated against live policy on the daemon at release.
- **Remote transport is the new attack surface.** Streamable HTTP binds to `127.0.0.1` by default; any non-local exposure is explicit operator opt-in and must sit behind an authenticated proxy. Even unauthenticated reachability cannot escalate privilege — the daemon independently enforces trust/policy/approval per request, and the server holds no secret — but an open endpoint could let an unauthorized local/remote caller *issue requests as this agent*, so the default-deny bind and the hardening note are mandatory. Document this prominently in the README.
- **Logging/redaction.** The server logs lifecycle and errors **secret-free**: never log args, env, tokens, or the `result`/`error.message`/`approval.summary` payloads at info level. Reuse the T113 `logAuditFailure` discipline (class + `dedup_key` only) for audit-tap failures. Transport errors are mapped via the existing `mapTransportError`/`mapDaemonError`, not logged raw.
- **Audit correlation.** Every serialized result carries `audit_ref` (structurally always present, `null` inner ids when the daemon has not returned them — never fabricated). The `withAudit` tap mirrors a non-secret subset to the queryable index; a Postgres outage is swallowed and never blocks a tool call or weakens the substrate truth.

## Testing Plan

- **Unit — generator (`tools.test.ts`):** lists exactly nine tools; names/descriptions/`inputSchema` equal the descriptors verbatim (deep-equal, no mutation/clone drift); the set equals `CANONICAL_M1_TOOLS` names; intersection with `FORBIDDEN_AUTHORITY_VERBS` is empty (AC1 + no-authority invariant).
- **Unit — serializer (`serialize.test.ts`):** a table over all five statuses asserting the `structuredContent`/`content`/`isError` mapping — **`isError` true only for `status:"error"`**; `awaiting_approval` carries `handle` + `approval` and is **not** `isError` (AC3); `denied` carries the denial `error.code` and is not `isError`; `audit_ref` always present; round-trips through `validateEnvelope`. Pure, no daemon.
- **Unit — dispatch (`dispatch.test.ts`):** each name routes to the right handler with the right `deps` subtype (fake `DaemonCall`); unknown tool name → `errored('not_found')` not a throw; `room` is taken from the context, never from input; `mx_delegate_tool` receives `DelegateDeps`.

## Risks and Open Questions

1. **New package name/location — confirm.** Proposed `@mx-loom/mcp` at `packages/mcp` (sibling of `claude`/`registry`/`audit`). Alternative: fold the generic MCP server into a shared `@mx-loom/binding` package. Recommendation: a dedicated `@mx-loom/mcp` (clean dependency story, matches T602 "publish the MCP server").
2. **Low-level `Server` vs high-level `McpServer`.** Recommendation: **low-level `Server` + `setRequestHandler`** so descriptor JSON Schema passes through verbatim (no Zod round-trip). Confirm the pinned `@modelcontextprotocol/sdk@1.29.0` low-level API exposes `inputSchema`/`outputSchema` passthrough and the `InMemoryTransport` test pair as assumed; adjust module paths to the resolved SDK version.
3. **`isError` semantics for `denied`.** This spec recommends `denied` is **not** `isError` (governance outcome, model must read it). Some MCP clients may surface only `isError` and ignore `structuredContent`, hiding a denial. Mitigation: the human-readable `content` text states the denial plainly. Confirm this is the desired model-facing behavior; the alternative (mark `denied` as `isError`) risks runtimes retrying/aborting instead of replanning.
4. **`outputSchema` + `structuredContent` validation.** MCP ≥ a certain protocol revision lets clients validate `structuredContent` against `outputSchema`. Several descriptors have *open* output objects, and the envelope (not the bare `result`) is what we put in `structuredContent`. Decision: advertise `outputSchema` as the descriptor's (open where applicable) and do **not** have the server validate — or omit `outputSchema` to avoid client-side validation friction. Recommendation: advertise but document that `structuredContent` is the *envelope*, not the bare `result`; confirm against the SDK's expectations.
5. **Session lifecycle & multiplexing.** One server process = one `MxSession` = one agent registration (design §7, 1:1). For the remote transport, decide whether concurrent HTTP clients share that single session (recommended for M1, single-tenant) or each gets its own — multi-session/tenant is explicitly M5. Confirm single-session for v1.
6. **Wire-shape assumptions inherited, not re-pinned.** `call.start` param names, `CallResponse` disposition vocabulary, `invocation.get`, and `audit_ref` field availability are still staged behind the handlers' `MXL_CONFORMANCE_TWO_DAEMON=1` round-trip. T109 must not green its live-round-trip ACs without a daemon; treat a daemon rejection as red.
7. **MCP request cancellation.** The SDK supports a `notifications/cancelled` for in-flight requests. v1 likely ignores it (the model cancels via the `mx_cancel` *tool*, which maps `invocation.cancel`). Confirm that deferring protocol-level cancellation is acceptable for M1, and document it.
8. **Audit default.** Default sink is `NullAuditSink` (audit off unless `MXL_AUDIT_PG=1`), so AC "audit rows present" is a T114 concern via the live path. Confirm the MCP server is the correct single chokepoint for the tap (it is, per T113's staging), and that the in-process Claude shim (T110) applies its own tap independently.
